Setting up a Lua plugin

Boilerplate Lua code for implementing a loadable plugin

This section assumes some familiarity with Lua, at least in the context of the Visionary Render script editor.

Introduction to Lua modules

Lua has a concept of a module, which is a container for a collection of related functionality that can be required (or included) by another Lua script.

There are a number of different ways to implement a Lua module. For Visionary Render plugins, any of the methods can be used, based on developer's personal preference. An overview of these methods can be read on the Lua website: http://lua-users.org/wiki/ModulesTutorial

Older modules used the module method where code had no direct access to global variables. Instead the author had to localise them before calling module (e.g. local print = print). This tends to make developing a module more difficult.

Our new recommended way to develop Lua modules is to make everything inside the module local, and explicitly expose the functions that should be available to the user of the module. This is demonstrated as part of the Lua plugin example code below.

Visionary Render Lua Plugins

A Lua plugin in Visionary Render has full access to everything that a normal event script can do, such as all of the `vr` functions.

To instruct Visionary Render to load a Lua plugin, the quickest way is to put the .lua file in Documents/Visionary Render/plugins/<lua file name>/<lua file name>.lua (e.g. plugins/testplugin/testplugin.lua)

A Lua plugin requires the following functions in order to be loadable by the plugin manager in Visionary Render:

  • init() - called once when the plugin is loaded, and should initialise any necessary objects such as VR Tree nodes or menu items, etc;

  • name() - should return a human-readable name for this plugin to be used to display this plugin in the plugin manager settings;

  • version() - should return the version number, which is also used in the plugin manager settings;

  • (optionally) cleanup() - called once when the plugin is unloaded, and should delete anything that init() put in the VR Tree.

Note

It is important that the cleanup function correctly removes things added by the plugin. For more information see Enabling / Disabling plugins at runtime.

  • testplugin.lua
-- define the plugin functions as local functions
local function name()
  return "Lua Plugin"
end

local function version()
  return "1.0.0"
end

local function init()
  print("Init ", name())
end

local function cleanup()
end

-- export the functions to the Lua state
return {
  name = name,
  version = version,
  init = init,
  cleanup = cleanup
}
This will result in a testplugin table in the global Lua state (named after the plugin file name), containing the four functions we exported.

To test the plugin, save it in the location mentioned above, and Launch Visionary Render. Open the Diagnostics Window (CTRL + D), go to the Log tab, and somewhere in here should be the line {LUA} Init Lua Plugin - this is the output from your first plugin. You can also see this in the lower pane of the Script Editor (CTRL + L).

Any functions exported by the plugin are directly accessible by Visionary Render, either in the Script Console, or in Event script. For example, run the following in the script console:

print(testplugin.name())

The output window will display the output of the name function:

{LUA} Lua Plugin

Important Considerations

Warning

It is important to only define functions and local variables in the plugin script files, and not to execute any code outside of these functions. This is because, when loading the plugin, it is first loaded into a separate sandbox for querying its name and version, in order to check for conflicts with other plugins before exposing it to the main application Lua environment. This sandbox does not have access to any VR Lua functions or anything else in the main application environment.

If you are using the same conventions as the example script above, it is important to remember that anything without a local prefix will appear in the global Lua state. This does mean that plugins may, if they wish, define global Lua functions (e.g. vrMyPluginFunction()), however this is not recommended as there is a greater chance of collision with existing global functions or other user defined globals inside scene events.
